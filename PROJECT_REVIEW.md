# Обзор проекта "Добрые Лапки" - Система управления приютом для животных

## 🎯 Что было реализовано

### Полнофункциональная веб-система для управления приютом животных

Разработана комплексная платформа, которая решает реальные задачи приюта: от учета животных и обработки заявок на усыновление до финансового планирования и аналитики. Система не просто хранит данные - она помогает принимать обоснованные решения на основе аналитики и прогнозов.

---

## 🛠️ Как это реализовано

### Backend: Flask + SQLite - Простота и эффективность

**Почему Flask?** Выбран легковесный фреймворк Flask вместо более тяжелых решений (Django) по нескольким причинам:
- **Гибкость**: полный контроль над архитектурой приложения
- **Простота**: минималистичный подход, легко понять и модифицировать
- **Производительность**: быстрая обработка запросов для небольшого-среднего трафика
- **Расширяемость**: легко добавлять новые функции без избыточного "магического" кода

**Реализация:**
```python
# Чистая архитектура с разделением ответственности
@app.route('/admin/analytics')
@login_required
def analytics_dashboard():
    # Проверка прав доступа
    if current_user.role != 'admin':
        return redirect(url_for('index'))
    
    # Сбор данных через функции-помощники
    financial_summary = get_financial_summary()
    forecast = forecast_costs(3)
    
    # Рендеринг шаблона с данными
    return render_template('analytics.html', ...)
```

**Почему SQLite?** Для проекта такого масштаба SQLite - идеальный выбор:
- **Встроенная БД**: не требует отдельного сервера
- **ACID-совместимость**: гарантия целостности данных
- **Простота развертывания**: один файл `shelter.db`
- **Достаточная производительность**: для сотен записей работает отлично

**Структура БД:** 8 взаимосвязанных таблиц с внешними ключами обеспечивают целостность данных и возможность сложных запросов с JOIN.

### Frontend: Современный стек без фреймворков

**Почему без React/Vue?** Для этого проекта выбрана классическая связка HTML/CSS/JS:
- **Быстрая разработка**: нет необходимости в сборке и компиляции
- **Простота поддержки**: любой разработчик поймет код
- **Меньше зависимостей**: быстрее загрузка страниц
- **Достаточная интерактивность**: Chart.js + Vanilla JS решают все задачи

**Реализация адаптивности:**

### Аналитика: Chart.js + Кастомные алгоритмы

**Почему Chart.js?** 
- **Легковесность**: всего 60KB минифицированного кода
- **Красивые графики**: из коробки выглядят профессионально
- **Интерактивность**: hover-эффекты, масштабирование
- **Адаптивность**: автоматически подстраивается под размер экрана

**Реализация динамических графиков:**
```javascript
// Загрузка данных через AJAX без перезагрузки страницы
fetch(`/api/analytics/donations-timeline?days=${days}`)
    .then(res => res.json())
    .then(data => {
        // Обновление графика в реальном времени
        donationsChart.data.labels = data.map(d => d.date);
        donationsChart.data.datasets[0].data = data.map(d => d.total);
        donationsChart.update();
    });
```

**Прогнозирование затрат:** Реализован собственный алгоритм линейной регрессии без использования numpy/pandas:
- Анализ исторических данных за 90 дней
- Расчет среднего дневного расхода
- Вычисление тренда методом наименьших квадратов
- Прогноз на 3 месяца вперед

**Зачем свой алгоритм?** 
- **Независимость**: не требует тяжелых библиотек
- **Прозрачность**: понятно, как работает прогноз
- **Контроль**: можно легко модифицировать под нужды проекта

## 🔐 Безопасность: Многоуровневая защита

### Хеширование паролей

**Инструмент:** Werkzeug (PBKDF2 с SHA-256)

**Почему PBKDF2?**
- **Стандарт безопасности**: рекомендован NIST
- **Защита от брутфорса**: медленное хеширование
- **Соль**: каждый пароль имеет уникальную соль
- **Итерации**: 260,000+ итераций делают взлом практически невозможным

**Реализация:**
```python
# При регистрации
password_hash = generate_password_hash(password)
# Результат: pbkdf2:sha256:260000$salt$hash

# При входе
if check_password_hash(user.password_hash, password):
    login_user(user)
```

### Защита от SQL-инъекций

**Проблема:** Прямая подстановка значений в SQL-запросы опасна:
```python
# ОПАСНО!
query = f"SELECT * FROM users WHERE username = '{username}'"
# Если username = "admin' OR '1'='1", получим доступ ко всем пользователям
```

**Решение:** Параметризованные запросы:
```python
# БЕЗОПАСНО
cursor.execute('SELECT * FROM users WHERE username = ?', (username,))
# SQLite автоматически экранирует специальные символы
```

**Зачем это важно?** SQL-инъекции - одна из самых распространенных атак. Параметризованные запросы - стандарт индустрии.

### Прогнозирование затрат

**Что реализовано:**
- Анализ исторических данных за 90 дней
- Расчет тренда методом линейной регрессии
- Прогноз расходов на 3 месяца вперед

**Алгоритм:**
1. Собираем данные о расходах за последние 90 дней
2. Вычисляем среднее дневное значение
3. Рассчитываем тренд (растут или падают расходы)
4. Прогнозируем: `прогноз = среднее * 30 + тренд * 30 * номер_месяца`

**Зачем прогнозирование?**
- **Планирование бюджета**: понимание будущих расходов
- **Привлечение средств**: можно показать спонсорам, сколько нужно
- **Оптимизация**: выявление трендов помогает оптимизировать расходы

**Реализация алгоритма:**
```python
# Линейная регрессия вручную (без numpy)
n = len(costs)
x_sum = sum(range(n))
y_sum = sum(costs)
xy_sum = sum(i * costs[i] for i in range(n))
x2_sum = sum(i * i for i in range(n))

# Формула: a = (n*Σxy - Σx*Σy) / (n*Σx² - (Σx)²)
trend = (n * xy_sum - x_sum * y_sum) / (n * x2_sum - x_sum * x_sum)
```

### Визуализация данных

**Инструмент:** Chart.js 4.4.0

**Типы графиков:**
1. **Линейный график** - динамика пожертвований во времени
2. **Столбчатая диаграмма** - тренды усыновлений по статусам
3. **Круговая диаграмма** - распределение инвентаря по категориям
4. **Линейный график прогноза** - будущие затраты

**Зачем визуализация?**
- **Понимание данных**: графики легче воспринимать, чем таблицы
- **Выявление трендов**: визуально видны закономерности
- **Презентация**: красивые графики для отчетов и презентаций

**Интерактивность:**

## 🗄️ База данных: Структура и связи

### Архитектура данных

**8 таблиц с четкой структурой:**

1. **users** - пользователи системы
   - Хранит учетные данные и роли
   - Основа для аутентификации

2. **animals** - животные в приюте
   - Полная информация о каждом животном
   - Связь с заявками и пожертвованиями

3. **adoption_requests** - заявки на усыновление
   - Foreign key → animals
   - Статусы: pending, approved, rejected

4. **donations** - пожертвования
   - Foreign key → animals (опционально)
   - Типы: на приют / на животное

5. **inventory** - инвентарь
   - Категории: food, medicine, supplies, other
   - Используется для расчета расходов

6. **news** - новости приюта
   - Публичная информация

7. **stray_animals** - бездомные животные на карте
   - Геолокация (lat, lon)

8. **logs** - логи действий
   - Foreign key → users
   - Аудит всех операций

**Зачем такая структура?**
- **Нормализация**: данные не дублируются
- **Целостность**: внешние ключи гарантируют корректность
- **Масштабируемость**: легко добавлять новые таблицы
- **Производительность**: индексы ускоряют запросы

## 🚀 API: RESTful подход

### Endpoints для аналитики

**GET /api/analytics/donations-timeline?days=30**
- Возвращает данные о пожертвованиях за период
- Формат: JSON
- Используется для построения графика

- Возвращает тренды усыновлений
- Разбивка по статусам
- Используется для столбчатой диаграммы

**GET /api/analytics/inventory**
- Анализ инвентаря по категориям
- Используется для круговой диаграммы

**Зачем API?**
- **Разделение логики**: данные отдельно от представления
- **Переиспользование**: можно использовать в мобильном приложении
- **Производительность**: загрузка только нужных данных
- **Масштабируемость**: легко добавить новые endpoints
